# 06-DEBUGGING — систематический дебаггинг

## Принцип: root-cause, не symptom-fix

Исправлять симптом без понимания причины = откладывать проблему. Каждый баг требует гипотезы, воспроизведения и доказательства исправления.

## Алгоритм (4 шага)

```
1. ВОСПРОИЗВЕСТИ
   Найди минимальный набор условий, при которых баг проявляется.
   Если не воспроизводится — значит, не понял проблему.

2. ИЗОЛИРОВАТЬ
   Сужай: слой? компонент? конкретная функция?
   Используй Grep/Read/Bash, не чтение всего кода.

3. ГИПОТЕЗА → ПРОВЕРКА
   Выдвини одну конкретную гипотезу.
   Напиши failing test ДО исправления.

4. ИСПРАВИТЬ + ЗАДОКУМЕНТИРОВАТЬ
   Фикс минимальный. Коммит с описанием причины:
   "fix(crawler): prevent None dereference when robots.txt missing
   Root cause: requests.get returns None on timeout if allow_redirects=False.
   Refs: P2"
```

## Когда спавнить `auditor` (reviewer)

| Ситуация | Действие |
|---|---|
| Баг в зоне инвариантов §6 (read-only граница, probe-geo, ПДн) | Немедленно `auditor`/Opus |
| Security-подозрение (утечка данных, обход auth) | `reviewer` с security-линзой + `auditor` |
| Баг воспроизводится в prod-логах, но не в тестах | `auditor` для adversarial-анализа |
| Непонятна root-cause после 30 мин изоляции | Эскалация в `03-ESCALATION.md` |
| Простая логическая ошибка (off-by-one, None check) | Сам; тест + фикс |

## Доказательства в коммитах

Коммит фикса должен содержать:
- **Root cause** — одна строка в теле коммита
- **Failing test** — добавлен ДО фикса (или в том же коммите)
- **Refs** — PNN + ADR если затронут инвариант

```
fix(scoring): cap readiness score to [0,100] range

Root cause: integer overflow when schema_coverage > 100 items.
Added test: tests/test_scoring.py::test_score_overflow_guard
Refs: P3, §6-invariant-2 (honest-forecast)
```

## Чек-лист по категориям WIZOR

**Crawl / probe ошибки:**
- Timeout настроен? User-agent выставлен?
- Probe к внешним LLM идёт не с RU-IP (инвариант §6.4)?
- N≥5 повторений для статистики (инвариант §6.7)?

**Score / LLM-вызовы:**
- llms.txt не влияет на score (инвариант §6.5)?
- Возвращается диапазон+доверие, а не точное число (инвариант §6.2)?
- Uncertainty задокументирован при шуме?

**Auto-fix / CMS-connector:**
- Операция идемпотентна?
- DPA акцептован ДО записи (инвариант §6.3)?
- Append-only audit log обновлён?
- Rollback протестирован?

**Multi-tenant:**
- `tenant_id` фильтрует все запросы (инвариант §6.8)?
- Нет cross-tenant утечки в логах?

## Anti-patterns

- **Фикс без теста** — следующий рефактор сломает снова.
- **Коммит «fix bug»** — без описания root-cause бесполезен для будущих агентов.
- **Читать весь codebase в поиске бага** — сначала Grep по симптому, потом Read нужного файла.
- **Игнорировать инварианты §6** — любой баг в их зоне требует `auditor`.
