## Context

Главный риск (H1 из Phase 0) — клиенты не готовы давать API-доступ к production-сайту. Если auto-fix — порог входа в продукт, и гипотеза не подтвердилась, значительные инженерные усилия потрачены впустую. Нужна последовательность фаз, при которой продукт монетизируется до подтверждения гипотезы, а тяжёлые CMS-коннекторы стартуют только при доказанном спросе.

## Decision

Роадмап **P0–P10 организован read-only-first**: все фазы, не требующие write-доступа (P1–P9), строятся независимо и монетизируются сами по себе. P10 (Auto track: WP-коннектор + auto-fix + trust-ladder + rollback + DPA) **явно gated_by P0** — стартует только после прохождения Phase 0 gate (5 гипотез H1–H5 подтверждены). Это превращает auto-fix из порога входа в **апгрейд** поверх работающего продукта.

Порядок P0–P10 зафиксирован в Charter §5 (таблица роадмапа) с колонками `depends_on` и `gated_by`. Авто-починка как upgrade-path = защита от риска, а не оправдание его.

## Consequences

- Продукт имеет ценность (Tier 0 PLG + Manual track) уже на P7–P9, до тяжёлой CMS-интеграции.
- P10 — управляемый de-risk: если H1/H5 не подтверждены, P10 не строится, но потеря невелика.
- evidence-gated scaling (PRD-доктрина) реализован архитектурно.
- Компромисс: auto-fix (главный дифференциатор) выходит позже; но это сознательный выбор в пользу «живого продукта с ценностью» vs «мёртвого дифференциатора».
- Phase 0 (Discovery) параллелит P1–P5 — ответы к gate нужны до P6, не до P10.

## Alternatives considered

| Альтернатива | Pro | Contra | Почему отклонили |
|---|---|---|---|
| Auto-fix с P1 | Дифференциатор раньше | Риск H1; напрасные инвестиции без доказанного спроса | Противоречит evidence-gated doctrine |
| Auto-fix только фаза C | Полное де-рискование | Задерживает moat на 18+ мес | Окно рынка 12–18 мес (PRD §2.4) не позволяет |
| Gate только по H1 | Проще условие | H5 (юр DPA-готовность) одинаково критична | Все 5 гипотез нужны для safe auto-apply |
