## Context

ADR-0017 дал полную внутри-фазовую автономию. Founder хочет, чтобы эта автономность была **подтверждена результатами проверок**, а не «на слово». Для продукта, чья ценность — реальные исходы ИИ-видимости, мок-юнит-тестов недостаточно: нужен живой end-to-end прогон против реальных сервисов («live-gold», по образцу ORIION) там, где это возможно.

## Decision

1. **Перед поднятием/мерджем PR ответственный агент ОБЯЗАН автономно прогнать и получить зелёный результат:**
   - **a. Авто-тесты** — unit + integration (по `conventions.md`: coverage ≥70% / ≥85% security-critical).
   - **b. Live-gold там, где возможно** — end-to-end прогон против РЕАЛЬНЫХ сервисов с малым golden-набором (funded-ключи, реальный тест-сайт/прокси), с проверкой ожидаемого вывода. «Где возможно» = существуют возможность + ключи/фикстуры.

2. **Evidence записывается и является входом гейта.** `verifier` (шаг 6) пишет `_session-context/VERIFY-<phase>-<ts>.md` с секцией `## Live-gold` (дата, что прогонялось, какие funded-ресурсы, pass-каунты, golden-диффы). `hard_thresholds[...].evidence_url` гейта указывает на него. Ни один гейт не закрывается без evidence.

3. **Никаких тихих пропусков.** Если live-gold **невозможен** (нет funded-ключей, до-инфра, внешняя зависимость недоступна), агент ОБЯЗАН: (a) прогнать максимум возможного (mock/replay/staging) и получить зелёный; (b) записать **явный DEFERRAL** в verify-report и гейт (`deferred_live_gold: {reason, what, founder_action}`); (c) НЕ помечать live-gold «выполнен». Founder видит deferral **на гейте**. («Где возможно» → явный deferral, когда нет; никогда не скрыто.)

4. **Условие авто-мерджа (amends ADR-0017 §2):** PR внутри фазы авто-мерджится на зелёный CI + `reviewer` APPROVE + `auditor` PASS + **тесты зелёные + live-gold PASS (или зафиксированный deferral)**. Шаг 6 (`verifier`) гейтит это: PR не поднимается, пока шаг 6 не зелёный.

5. **Live-gold ограничен:** малый канонический golden-набор; стоимость считается в cost-budget (ADR-0003); funded-ключи/фикстуры — от founder (`PLACEHOLDERS.md`). На фазу `planner` объявляет live-gold-scope в `PLAN.md`; `verifier` исполняет; exit-gate phase-spec'а называет live-gold-критерий там, где фаза касается внешних сервисов.

6. **Live-gold по треку:**
   - **read-only** (P2/P3/P5/P6/P7/P8) — живой crawl/probe golden против реальных URL/моделей (probe соблюдает N≥5+CI, ADR-0016).
   - **infra** (P1/P4/P9) — живой smoke против реальных Postgres/Redis/Keycloak (testcontainers/staging).
   - **auto** (P10) — применение на **founder-owned тест-WP-сайт** (наша фикстура, НЕ клиент → внутри dev-автономии; реальный клиентский prod остаётся под ADR-0015/0017), живая проверка идемпотентности + rollback.

## Consequences

- **+** Каждая фаза само-доказывается ДО того, как founder вообще смотрит → ревью на гейте идёт по подтверждённым результатам, не по обещаниям. Реализует «автономность, подтверждённую результатами».
- **+** Ловит реальные сбои (дрейф API, гео/прокси, отказ schema), которые моки пропускают — критично для этого продукта.
- **−** Live-gold стоит реальных денег/времени и требует funded-ключей рано → митигируется малым golden-набором, cost-cap'ами, явным deferral при отсутствии ключей.
- **−** Часть фаз не может полноценно live-тестироваться до поздней инфры → no-silent-skip deferral, видимый на гейте.

## Alternatives considered

- **Только моки/CI перед PR** — отклонено: не доказывает реальные исходы; продукт про живую ИИ-видимость.
- **Блокировать фазу, пока live-gold невозможен** — отклонено: стопорит легитимно-не-готовые фазы; «там, где возможно» = deferral, не блок.
- **Live-gold только на гейте вручную founder'ом** — отклонено: противоречит автономности (ADR-0017); агент делает сам.
