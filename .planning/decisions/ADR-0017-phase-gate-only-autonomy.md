## Context

Founder хочет получать на ревью **только гейты фаз**. Всё внутри фазы агенты выполняют сами — включая ревью, мердж PR, аудит, итерации, **запуск дополнительных сессий Claude Code** для отдельных задач и управления контекстным окном. Это меняет ADR-0009, где founder был per-PR-апрувером для tier 3+.

## Decision

1. **Human-in-the-loop = только гейт фазы.** Единственная штатная точка ревью человеком — exit-гейт фазы (`gates/*.md`, поле `founder_signature`). Внутри фазы агенты планируют, реализуют, ревьюят, верифицируют, аудируют и **мерджат PR автономно** — без per-PR аппрува человека. На гейте founder смотрит кумулятивный результат фазы (hard_thresholds гейта + вердикт аудита + deliverables), а не отдельные PR.

2. **Условие авто-мерджа PR внутри фазы:** зелёный CI **+** `reviewer` = APPROVE **+** `auditor` = PASS / PASS-WITH-FIXES (фиксы применены). Человеческая колонка tier-таблицы (ADR-0009) **упраздняется** в пользу гейта. Tier по-прежнему задаёт глубину аудита (ADR-0005) и строгость CI, но не маршрутизирует на человека.

3. **Автономные доп. сессии.** Оркестратор (главная сессия) и суб-агенты МОГУТ запускать дополнительные сессии Claude Code для: (a) параллельного исполнения независимых задач; (b) управления контекстным окном (handoff + продолжение в свежей сессии при приближении к бюджету charter §9). Механизмы: основной — Agent tool (Task), свежий контекст; для полностью независимой сессии — headless `claude -p` (non-interactive) через Bash. Каждой порождённой единице даётся scoped-промпт + минимальный JIT-контекст (charter §9).

4. **Guardrails автономии (чтобы «сама» оставалась безопасной и ограниченной):**
   - cost-budget kill-switch (ADR-0003): per-task hard $1.50, per-day $50, per-month $300 — автономный расход жёстко ограничен.
   - stagnation-kill: 30 мин без прогресса → авто-стоп.
   - глубина спавна сессий ≤ 2 (порождённая сессия не спавнит рекурсивно дальше 2 уровня); одновременных сессий ≤ 8.
   - каждая единица пишет handoff (charter §8.6); `memory-curator` — по-прежнему единственный писатель состояния/памяти.
   - граница автономии = scope активной фазы (`PLAN.md`); кросс-фазовая или вне-scope работа требует новой фазы/гейта.

5. **Эскалация — ИСКЛЮЧЕНИЕ, а не штатное ревью.** Агенты эскалируют к founder внутри фазы ТОЛЬКО при: (a) реально блокирующей неоднозначности, нерешаемой из spec/charter/контрактов; (b) решениях территории founder (юр/152-ФЗ-стратегия, деньги/прайсинг, бренд, фундаментальный архитектурный/стратегический разворот). Рутинное «проверь мою работу» НИКОГДА не идёт к founder — оно идёт к `reviewer`/`auditor`. Эскалации всплывают асинхронно (флаг в HANDOFF), это не «ревью».

6. **Граница: необратимые внешние действия остаются гейтнутыми даже внутри фазы** (НЕ покрыты грантом dev-автономии): применение правок на РЕАЛЬНЫЙ чужой/клиентский production-сайт; реальное движение денег / живые списания биллинга; подписание юр-соглашений (DPA) от лица компании; внешние коммуникации клиентам. Это **product-runtime** действия, управляемые trust-ladder + DPA продукта (ADR-0015) и opt-in конечного пользователя — это не «dev-ревью» и вне scope данного гранта. При разработке против фикстур/staging агенты полностью автономны; касание реальных внешних систем требует продуктовых consent-гейтов (заданы founder через GTM, не per-task dev-ревью).

## Consequences

- **+** Founder ревьюит ~11 гейтов за Phase A вместо десятков PR → радикально меньше прерываний, прямо отвечает директиве автономии.
- **+** Выше пропускная способность: агенты сами параллелятся и сами управляют контекстом.
- **−** Меньше гранулярного надзора внутри фазы → митигируется обязательным `auditor` (шаг 7) + стоячими инвариантами (charter §6) + подписью гейта + kill-switch'ами (cost/stagnation) + полной git-историей (всё откатываемо).
- **−** Повышается цена качества гейтов: hard_thresholds должны быть осмысленными — теперь это главная точка контроля человеком.

## Alternatives considered

- **Оставить per-PR tier-3+ аппрув (ADR-0009 как был)** — отклонено: founder явно хочет ревью только на гейтах.
- **Полная автономия включая внешние необратимые действия** — отклонено: outward-facing необратимые акты требуют consent (trust-ladder/DPA), и операционный принцип требует подтверждения для трудно-обратимых внешних действий.
